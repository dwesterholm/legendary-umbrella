<objective>
Recover three of the five "fields Booli lacks" — floor/våning, balcony, and BRF name — directly from the Apollo `Listing:<id>` entity the Phase-5 owned client already fetches, requiring ZERO new network call. Extend the internal listing model (`listingDataSchema` / `NormalizedListing`) with all five new fields as additive `.nullable()` fields (no migration — persisted into the existing JSONB `listing_data` column), and define the `Sourced<T>` provenance contract that Plan 02's gap-fill merge will populate.

Purpose: RESEARCH Pattern 1 ("read what's already there before fetching more") + Pitfall 1 — the ROADMAP's assumption that these fields require a broker page is FALSE against the real Phase-5 fixture. `floor` is inline (`{raw: 3}`), `brfName` is the final `/bostadsrattsforening/` breadcrumb label, and `balcony` is encoded in the `amenities` ref key. Shipping these as a pure code change removes 3 fields' worth of broker-CMS-coverage risk entirely and de-risks the phase. This is the interface-first plan: it defines the schema fields and `Sourced<T>` type that Plans 02 and 03 build against, so those executors receive the contract rather than reverse-engineering it.

Output: floor/balcony/brfName populated from Apollo in `client.ts`; five new nullable fields + `Sourced<T>` in `listing.ts`; fixture-backed tests.
</objective>
