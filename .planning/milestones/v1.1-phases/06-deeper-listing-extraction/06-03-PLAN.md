<objective>
Wire the broker-enrichment module into the primary flow and surface the five recovered fields in the report. `analyzeUrl` gains a second, optional enrichment step: after `fetchListing` succeeds, call `fetchBrokerListingPage(agencyListingUrl)` in its OWN try/catch that can only ADD fields — never fail the primary analysis (RESEARCH Pattern 4, inverted from `fetchSoldComps`'s rethrow). The merged, provenance-tagged fields persist into the existing JSONB `listing_data` column (no migration). `ListingSummary` renders the five fields per the UI-SPEC contract with provenance captions and an honest "Ej tillgänglig" / soft-banner degradation.

Purpose: LSTG-04's central success criterion — "when the broker page is unreachable or unparseable, the primary listing analysis still succeeds exactly as before, with the missing fields shown as unavailable rather than fabricated." This is also the phase's only UI surface (UI-SPEC): purely additive to `ListingSummary`, no new routes/flows. This plan closes the loop: Plan 01 gave the Apollo fields + schema, Plan 02 gave the guarded broker module, this plan connects them to persistence and the user.

Output: `analyze.ts` broker-enrichment step + merge fold; `analyze.test.ts` (first-ever integration test for analyzeUrl); `listing-summary.tsx` five-field render with provenance + degradation.
</objective>
