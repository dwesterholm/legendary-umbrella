# Phase 10 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Scope note:** This phase introduces NO new visual language. It extends the exact same warm-earthy design system (warm-gray / sage / terracotta OKLCH palette, shadcn `new-york` primitives, DM Sans, Lucide icons) established in Phases 1–9. Every token, component, and copy pattern below is either a direct citation of an existing file (`discovery-candidate-card.tsx`, `discovery-progress.tsx`, `report-flags.tsx`, `brf-score-card.tsx`) or a same-vocabulary extension of one. This phase touches ONLY the `/discover/[jobId]` results surface — no new route, no new page shell. Everything here lives behind the `DISCOVERY_ENABLED` flag (Phase 9's Feature Flag Contract applies unchanged; not restated here).

## Color

Identical OKLCH tokens from `src/app/globals.css` — no new colors:

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `--color-warm-white` / `--background` (oklch(0.98 0.005 80)) | Page background (unchanged from Phase 9) |
| Secondary (30%) | `--color-warm-gray-50/100` + white `Card` surfaces | Candidate cards, signal-chip background (`bg-warm-gray-50`, mirrors `ReportFlags` list-item background) |
| Accent (10%) | `--color-sage-600` (oklch(0.50 0.08 150)) | **Reserved for:** the rank-position badge on each card (e.g. "#1"), the active/selected niche option, the hover border on candidate cards (`group-hover:border-sage-200`, unchanged from Phase 9) |
| Destructive / warning | `--color-terracotta-600` (oklch(0.50 0.14 35)) | Degenerate-ranking soft-warning banners only (too-small candidate set, no distinguishing signal) — same soft-warning vocabulary as Phase 7–9's `bg-terracotta-50`/`text-terracotta-600` banners |

Accent reserved for: (1) the numeric rank badge ("#1", "#2"...) shown on each ranked candidate card — sage, not a neutral gray, so rank reads as the one piece of "this phase's new information" on the card; (2) the currently-selected niche in the niche selector; (3) the existing candidate-card hover border (unchanged). Accent is NEVER used on the cited-signal chips themselves — those reuse `ReportFlags`' own severity-neutral `bg-warm-gray-50` chip treatment (signals are explanatory text, not a green/red verdict) — and NEVER on a numeric score, because no numeric score is ever displayed (binding constraint, see Copywriting Contract). Terracotta is reserved for degenerate/empty ranking states only, never for a "bad fit" candidate — a candidate with no niche signal is a data-limitation, not a red flag on the candidate itself.

| Element | Copy |
|---------|------|
| Primary CTA | N/A — no new submit action this phase. The niche selector re-orders in place (no submit button, no page reload); selecting an option is the "action." |
| Niche selector label | "Sortera efter" |

| Cited-signal chip (per candidate, per niche) | Reuses `ReportFlags`' exact chip vocabulary: a short Swedish label (e.g. "Låg pris/kvm vs området", "BRF: stambyte planerat, föreningen bekostar", "Nyligen renoverat kök" pending Phase 6 data) + "Källa: {sourceLabel}" caption, same `text-xs text-warm-gray-500` treatment as `report-flags.tsx` |
| No-signal-for-this-niche caption (per candidate) | "Inga tydliga signaler för denna sortering" — rendered where the signal-chip list would be, in the same `text-warm-gray-500 italic` treatment `BrfScoreCard` uses for "Ej tillganglig", NOT a warning/terracotta treatment (a candidate lacking a signal is not a defect, just uninformative for this niche) |
| Degenerate-ranking banner (too few candidates to rank meaningfully) | "För få träffar för att rangordna meningsfullt — visar dem i ursprunglig ordning." — terracotta soft-warning banner (`bg-terracotta-50`/`text-terracotta-600`, same shell as Phase 9's cap-reached/kill-switch banners); niche selector remains visible but this message replaces the reordering effect explanation |
| Ranking explanation sub-line (always visible when a niche is active) | "Rangordnat efter: {niche label}. Ordningen baseras på tecken i annonsen — ingen förutsägelse." — sits directly under the niche selector, same `text-sm text-warm-gray-500` treatment as Phase 9's `querySummary` sub-line under the page h1 |
| Empty state (zero candidates — inherited from Phase 9, unchanged) | "Inga träffar denna gång" / "Vi hittade inga annonser som matchade din beskrivning och dina filter. Prova att bredda sökningen eller ta bort ett filter." — ranking UI does not render at all when there are zero candidates (niche selector is hidden, not disabled, matching the "full absence over disabled state" precedent from the Feature Flag Contract) |
| Error state (ranking computation fails/throws) | Same terracotta banner shell: "Rangordningen kunde inte beräknas just nu. Kandidaterna visas i ursprunglig ordning." — degrades to the Phase 9 unranked list rather than blocking the whole results view; ranking is additive, never load-bearing for seeing results at all |
| Destructive actions | None exist in this phase. Selecting a different niche is non-destructive (always reversible, no confirmation needed) — reordering, not deleting. |

**No prediction/verdict framing** (binding constraint, carried from ROADMAP Phase 10 + REQUIREMENTS DISC-03 + the objective): copy never says a candidate "is the best match," "will need renovation," or "is a good deal" — only that specific, cited signals were found (e.g. "Låg pris/kvm vs området" states a fact, not a verdict). The word "rangordning"/"sortera" (ranking/sort) is used, never "matchning"/"poäng" (matching/score) in any user-facing copy, to keep the fit-to-intent framing distinct from a scoring/prediction framing.

## Component Inventory (per UI surface)

### 1. Niche selector (new — sits above the candidate grid on `/discover/[jobId]`)

- **Component:** shadcn `Select` (already in `src/components/ui/select.tsx`, same contract as Phase 9's "Antal rum" filter), NOT a new segmented-control/tabs primitive (see Design System discretion above).
- **Placement:** directly below the existing page heading + `querySummary` sub-line (`h1` "Sökresultat" / query recap), above the candidate grid. `flex items-center gap-3` row: label "Sortera efter" (`text-xs font-medium uppercase tracking-wider text-warm-gray-500`, matches Phase 9's filter-label treatment) + the `Select` control (`w-64`, matches the width of Phase 9's other filter selects).
- **Options (fixed order):** "Ingen (som hittad)" (default/neutral, selected on first load — preserves the Phase 9 scan order as the honest baseline, never auto-selects a niche), "Renoveringspotential", "Inflyttningsklar", "Stambyte planerat — föreningen betalar".
- **Behavior:** selecting an option client-side re-sorts the already-fetched candidate array in place (no new server round-trip, no page reload, no loading spinner — ranking is a pure client-side reorder over data already on the page per REQUIREMENTS DISC-03/ROADMAP "final pass over the job's candidates"). The ranking explanation sub-line updates immediately alongside the reorder.
- **Visibility:** hidden entirely (not disabled) when `candidates.length === 0` — matches the empty-state precedent (no selector to sort an empty list).
- **Degenerate case:** when the candidate set is too small to rank meaningfully (see Empty/Degenerate States below), the selector remains visible and interactive, but the sub-line copy switches to the degenerate-ranking banner text instead of the normal "Rangordnat efter: {niche}" line.

### 2. Ranked candidate card (extends `DiscoveryCandidateCard`, Phase 9)

- **Base:** same `Card`/`Link` shell as `discovery-candidate-card.tsx` — `border-warm-gray-200 bg-warm-white`, `group-hover:shadow-md group-hover:border-sage-200`, same address/price/3-column metrics row, same "Källa: Booli" + "Se full analys" footer row. Nothing existing is removed or restyled.
- **New element — rank badge:** a small sage badge in the `CardHeader`, positioned top-left before/beside the address `CardTitle` (`flex items-center gap-2`), e.g. a `Badge` with `bg-sage-600 text-white` showing "#{n}" (1-indexed position in the current niche's ordering). Only rendered when a niche other than "Ingen (som hittad)" is selected — the neutral/no-niche default shows NO rank badge (it isn't a ranking, it's the original scan order, and showing "#1"..."#N" there would misleadingly imply a ranking exists when none was requested).
- **New element — cited-signal list:** a new block appended below the existing metrics row, above the "Källa: Booli" footer. Reuses `ReportFlags`' exact chip presentation (`rounded-lg border border-warm-gray-100 bg-warm-gray-50 p-3`, `Badge variant="secondary"` per signal + "Källa: {sourceLabel}" caption in `text-xs text-warm-gray-500`) — this phase does NOT invent a new chip style; it literally imports/adapts the same visual list `ReportFlags` renders. Each signal chip's label is drawn from a niche-specific vocabulary (see Copywriting Contract), never a raw numeric field name. When multiple signals contributed to a candidate's rank, show up to 3 (matches `ReportFlags`' un-truncated-but-reasonable list length in existing report contexts) — if more exist, do not silently truncate the count the user sees drive the rank without at least summarizing ("+{n} till" is out of scope for v1; cap the shown set at what's meaningful and let the planner size it during implementation).
- **No-signal state (per card, per niche):** if a candidate has zero cited signals for the currently active niche, render the italic "Inga tydliga signaler för denna sortering" caption in place of the signal-chip block — same treatment, same position, so the card's layout height doesn't visually jump between candidates that do/don't have signals.
- **What is explicitly NOT added to the card:** no numeric score, no percentage match, no star rating, no color-coded "good/bad" badge on the card itself (binding constraint from ROADMAP/CONTEXT — "NOT an opaque numeric score front-and-center"). The rank badge is an ordinal position, not a score; it carries no color semantics beyond the single sage accent already used for "this is new/notable information," identical across every rank position (no red/green gradient by rank).

### 3. Empty/degenerate states (new)

- **Zero candidates (inherited unchanged from Phase 9):** existing dashed-border empty-state block renders exactly as today; the niche selector does not render at all in this case (see Component 1 above).
- **Candidate set too small to rank meaningfully** (e.g. 1–2 candidates, or ROADMAP's kill-criterion-adjacent "not enough data to distinguish"): niche selector stays visible and functional, but:
  - The ranking explanation sub-line is replaced with the degenerate-ranking banner copy ("För få träffar för att rangordna meningsfullt — visar dem i ursprunglig ordning."), styled as the same terracotta soft-warning banner (`rounded-lg bg-terracotta-50 px-4 py-3`, `text-sm text-terracotta-600`) used for Phase 9's cap-reached/kill-switch messages — NOT a full-page block, just an inline banner between the selector and the grid.
  - Candidates still render in the grid, in their original (unranked) order — no rank badges shown in this state, since there is no meaningful ranking to badge.
  - This state is a presentation decision over the SAME candidate array Phase 9 already produced — it requires no new data fetch, no new job status, and does not block viewing the (unranked) results.
- **No distinguishing signal for a given candidate under the selected niche** (a normal-sized candidate set, but one candidate has nothing to cite for the active niche): handled per-card via the "Inga tydliga signaler för denna sortering" caption (see Component 2) — this is NOT a page-level degenerate state, it's a per-card one, and it does not block that candidate from still appearing at whatever rank position its (possibly zero) score placed it.
- **Ranking computation error** (deterministic ranking function throws/returns malformed data — CR-01-style defense): the whole results view falls back to Phase 9's unranked candidate grid with the error-state banner copy from the Copywriting Contract ("Rangordningen kunde inte beräknas just nu...") rendered where the ranking sub-line would be. This mirrors the existing codebase pattern of independent degradation (a broken enrichment layer never blocks the base data it's layered on top of — see LSTG-04's broker-fetch-never-fails-primary-analysis precedent).

### 4. Cross-cutting: rank stability + visible reordering

- **Visible reorder requirement (ROADMAP success criterion 1):** switching the `Select` value must visibly change the DOM order of cards in the grid — implement as a straightforward re-sort-and-re-render of the candidate array (React key on a stable candidate identity, e.g. `sourceListingUrl` or array index pre-sort, so React doesn't just relabel positions without an actual visual shuffle). No animation is required by this spec (DISCRETION: a plain instant reorder is acceptable; a future polish pass MAY add a `transition` if the planner/executor judges it low-risk, but it is not a requirement here).
- **Grid layout unchanged:** same `grid grid-cols-1 gap-4 md:grid-cols-2 lg:grid-cols-3` as Phase 9 — ranking changes card ORDER within the existing grid, never the grid's column/gap structure.
