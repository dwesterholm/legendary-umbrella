# Phase 11 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Scope note:** This phase introduces NO new visual language for the base design system — same warm-earthy OKLCH palette, shadcn `new-york` primitives, DM Sans, Lucide icons established in Phases 1–10. It DOES introduce one deliberately new, deliberately DIFFERENT visual treatment: the "AI-bedömning av bilder" vision section, whose entire job is to look and feel unlike `ReportFlags`/`BrfScoreCard` deterministic chips so a user can tell interpreted-from-images apart from verified-from-documents at a glance (ROADMAP Phase 11 success criterion 2, REQUIREMENTS DISC-04). This phase touches the candidate/analysis view only, behind `DISCOVERY_ENABLED` (Phase 9's Feature Flag Contract applies unchanged — full absence when off, never a disabled/greyed-out state).

## Color

Identical OKLCH tokens from `src/app/globals.css` — no new colors added. The critical design decision this phase makes is which EXISTING tokens the vision section uses vs. avoids, to guarantee it never looks like a deterministic flag:

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `--color-warm-white` / `--background` | Page background (unchanged) |
| Secondary (30%) | `--color-warm-gray-50/100` | Deterministic-flag chips, candidate/BRF cards (unchanged — `ReportFlags`/`BrfScoreCard` keep their existing look verbatim) |
| Vision section surface (NEW usage, existing token) | `--color-warm-gray-100` border + `--color-warm-white` fill, NOT `--color-warm-gray-50` | The vision card's own outer shell — one step lighter/more neutral than a flag chip's `bg-warm-gray-50`, with a slightly heavier `border-warm-gray-200` (vs. flags' `border-warm-gray-100`) so it reads as a distinct "frame," not a chip |
| Accent (10%) | `--color-sage-600` | Reserved for: rank badges (Phase 10, unchanged), the active/selected niche option (Phase 10, unchanged). **The vision section NEVER uses sage.** A hedged image-interpreted claim is not a positive/verified signal — using the "trust" accent color on it would contradict the hedge. |
| Vision marker accent (NEW, reuses existing terracotta-50/600, NOT a new hue) | `--color-terracotta-50` background wash + `--color-terracotta-600` icon/text | The `Eye` icon badge and the "från bildtolkning" marker chip. Terracotta is already the codebase's "worth a second look / soft caveat" vocabulary (staleness captions, degenerate-ranking banners, broker-fetch-failed banners) — reusing it here (rather than sage/green or destructive/red) correctly signals "interpreted, treat with care" without inventing a new hue that would need its own trust-language onboarding |
| Destructive / warning | `--color-destructive` | UNUSED by vision claims (a hedged image interpretation is never a red/destructive flag — that vocabulary is reserved for deterministic BRF/financial red flags only) |

**Accent reserved for (explicit list, binding):**
1. Sage (`sage-600`/`sage-100`) — deterministic-flag "green" severity, rank badges, selected niche. NEVER appears in the vision section.
2. Terracotta (`terracotta-50/100/600`) — (a) deterministic soft-warning banners (staleness, degenerate ranking, broker-fetch-failed — unchanged), AND (b) the vision section's own identity marker (icon badge, "från bildtolkning" chip, section header underline). This is a deliberate DUAL use of the existing terracotta vocabulary — both meanings are "soft caveat, not a hard failure," so sharing the hue is consistent with the existing color language rather than diluting it. The two uses are still distinguishable by icon (Eye vs. no icon) + copy + position (vision section is spatially separate, per Component Inventory).
3. Destructive (`--color-destructive`) — deterministic red-severity flags only (`ReportFlags`' `severityChip`), and the "Osäker — kontrollera själv" BRF badge. NEVER used for a vision claim, no matter how low-confidence — a low-confidence vision claim is suppressed entirely (see Copywriting Contract), never shown in red.

**Why the vision section must NOT reuse `bg-warm-gray-50`/`border-warm-gray-100` (the flag-chip look):** `ReportFlags` list items (`rounded-lg border border-warm-gray-100 bg-warm-gray-50 p-3`) and the vision claim cards must be visually distinguishable even in grayscale/at-a-glance, per ROADMAP success criterion 2 ("a user can tell at a glance which claims are image-interpreted vs verified"). The vision section therefore uses a one-step-different neutral pairing (`border-warm-gray-200` / `bg-warm-white`) PLUS the terracotta `Eye`-icon header, so the difference survives even if a user only skims color, only skims icons, or only skims copy.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Vision section title | **"AI-bedömning av bilder — kan vara fel"** (LOCKED verbatim, from ROADMAP/CONTEXT) |
| Vision section sub-label (always visible, directly under the title) | "Tolkat från bilder i annonsen — inte en verifierad uppgift. Kontrollera själv innan du drar slutsatser." |
| Per-claim hedge phrasing (examples, pattern not exhaustive) | "Köket verkar renoverat", "Badrummet ser omodernt ut", "Allmänt skick verkar välskött" — always a hedging verb ("verkar", "ser ut att", "tyder på"), NEVER "är" (is) / NEVER a bare adjective without a hedge verb. Banned words in any vision claim: "garanterat", "definitivt", "kommer att", "bör köpas/inte köpas" |
| Per-claim image citation | **"Bild {n}"** next to a small thumbnail crop of the cited source image (see Component Inventory §2) — e.g. "Bild 3" beside a 48×48px thumbnail. Never a bare claim with no citation — a claim with no resolvable source image is not rendered at all (fails closed, mirrors `ReportFlags`' "no uncited flag" discipline) |
| "från bildtolkning" marker (wherever a vision signal appears near ranking/score context) | **"från bildtolkning"** — a small terracotta outline `Badge` (not filled — filled terracotta is reserved for the vision-section header itself) placed immediately beside any vision-derived value that appears outside the vision section's own card (e.g. if a future niche or ranking context ever surfaces a vision signal). Phase 11 itself does NOT feed vision into ranking (binding constraint — REQUIREMENTS "Out of Scope" + ROADMAP criterion 3), so in practice this marker currently has no live call site; it is specified now so any future integration has zero ambiguity about the required treatment. |
| No-gallery empty state | "Inga bilder tillgängliga för den här annonsen — ingen bildbedömning kunde göras." — rendered in place of the whole vision section content, same `italic text-warm-gray-500` treatment as `BrfScoreCard`'s "Ej tillganglig", inside the SAME distinct vision-section shell (so its absence is still framed, not a silently missing block) |
| Vision skipped — cost cap | "Bildbedömning kördes inte för den här annonsen (sökgränsen för bildanalys nåddes)." — same shell + treatment as the no-gallery state; distinguishes "we chose not to look" from "there was nothing to look at" |
| Low-confidence — suppressed | "För osäkert för att visa — inga bildbaserade slutsatser kunde dras med rimlig säkerhet." — shown ONLY when every candidate claim for this listing was suppressed for low confidence (i.e. the section would otherwise be empty); an individual low-confidence claim is simply dropped from the list (never shown greyed-out or with a "low confidence" badge — showing a hedged-hedge is more confusing, not more honest) |
| Section-level honest disclaimer (always present, everywhere the section renders with content) | "Kan vara fel — dessa bedömningar är AI:ns tolkning av bilder, inte en besiktning." — sits at the bottom of the vision card, `text-xs text-warm-gray-500`, mirrors `BrfScoreCard`'s closing "Betyget beräknas i kod..." explainer-line pattern |
| Primary CTA | N/A — no new submit action. The vision section is read-only presentation woven into the existing candidate/analysis view; there is no button this phase introduces. |
| Destructive actions | None exist in this phase. |

**No verdict framing** (binding constraint, carried from ROADMAP/REQUIREMENTS/CONTEXT): copy never says a condition "is good/bad," never recommends "buy/don't buy," never assigns a numeric or letter grade to condition. Only hedged, cited observations. The word "bedömning" (assessment) is used only in the fixed section title (part of the locked string) — per-claim copy avoids repeating "bedömning"/"betyg" so it doesn't accidentally borrow the BRF score's authority language.

## Component Inventory (per UI surface)

### 1. Vision section shell — "AI-bedömning av bilder" (new component, e.g. `gallery-condition-vision.tsx`)

- **Structural position:** renders as its OWN `Card`, spatially separate from any `ReportFlags`/`BrfScoreCard` block — minimum `space-y-6` (24px, the `lg` token) between this card and any deterministic-flags block on the same page. On the candidate/analysis view, place it AFTER the deterministic BRF/price sections, never interleaved between them, so a user's reading order is "verified facts first, then hedged image interpretation."
- **Visual shell:** `Card` with `border-warm-gray-200 bg-warm-white` (NOT `border-warm-gray-100`/`bg-warm-gray-50` — see Color rationale). `CardHeader` contains: a terracotta-tinted icon badge (`flex h-8 w-8 items-center justify-center rounded-full bg-terracotta-50`, `<Eye className="h-4 w-4 text-terracotta-600" />` from `lucide-react`) beside the `CardTitle` "AI-bedömning av bilder — kan vara fel" (`text-base font-semibold text-warm-gray-900` — one step down from `BrfScoreCard`'s `text-2xl`, since this is a sub-section of the analysis view, not the page's primary heading), then the sub-label paragraph directly below in `text-sm text-warm-gray-500`.
- **What is explicitly NOT reused from `ReportFlags`:** the `Badge variant="secondary"` + `severityChip()` color-coded chip treatment, the `rounded-lg border border-warm-gray-100 bg-warm-gray-50 p-3` list-item shell, and the `blockquote` verbatim-quote treatment are all deliberately NOT reused here — reusing any of them would defeat the at-a-glance distinction this phase exists to create.
- **Why `Eye` (lucide-react):** communicates "this is what the AI looked at / saw," distinct from `ReportFlags`' text-only badges and `BrfScoreCard`'s numeric grade badge — no other component in the codebase currently uses an icon as its primary identity marker, making this a clean, unambiguous new signifier reserved exclusively for image-interpreted content. Do not reuse `Eye` anywhere else in the app (icon-to-meaning must stay 1:1).

### 2. Condition-claim row (per attribute, e.g. kitchen/bathroom/overall)

- **Shell:** each claim is its own row inside the vision `CardContent`, `flex items-start gap-3 rounded-lg border border-warm-gray-100 bg-warm-white p-3` — note `bg-warm-white`, NOT `bg-warm-gray-50` (further visual distinction from a `ReportFlags` list item, which uses `bg-warm-gray-50`).
- **Left: image citation thumbnail.** A `48×48px` (`h-12 w-12`) rounded (`rounded-md`) crop of the cited gallery/floor-plan image, `object-cover`, with a small overlaid or adjacent caption **"Bild {n}"** (`text-xs text-warm-gray-500`, positioned directly under or beside the thumbnail). If the thumbnail URL is unavailable (e.g. failed to persist), fall back to a plain "Bild {n}" text label with no image (never a broken-image icon) — same "degrade the visual, keep the citation" discipline as `ReportFlags`' pageRef-optional caption.
- **Right: the hedged claim text.** `text-sm text-warm-gray-700` (matches `ReportFlags`' `blockquote` body color, but NOT italicized/NOT a blockquote — this is the AI's own hedged statement, not a verbatim source quote, so it should not visually claim to be a direct citation the way `sourceQuote` blockquotes do elsewhere).
- **Attribute label (small, above the claim text):** `text-xs font-medium uppercase tracking-wider text-warm-gray-500` — e.g. "KÖK", "BADRUM", "ALLMÄNT SKICK" — same label treatment as `BrfScoreCard`'s `METRIC_META` labels, for consistency of "what am I looking at" affordance, without borrowing the metric CARD shell itself.
- **No confidence badge, no severity badge, no color-coded rating** on this row — the ENTIRE row carries exactly one visual signal (the terracotta section header above it) that this is interpreted evidence; repeating a confidence/severity badge per-row would visually converge back toward the `ReportFlags` pattern this section must avoid.

### 3. Degraded / empty / cost-capped states (new)

All four states below render INSIDE the same vision-section shell (icon + title + sub-label always present) so the section's identity is consistent whether or not it has content — never a silently absent section when `DISCOVERY_ENABLED` is on and a listing was eligible for vision.

- **No gallery available:** copy per Copywriting Contract ("Inga bilder tillgängliga…"), rendered as the sole `CardContent` in place of claim rows, `italic text-warm-gray-500`, `text-sm`.
- **Vision skipped due to cost cap:** copy per Copywriting Contract ("Bildbedömning kördes inte…"), same treatment. This state must be DISTINGUISHABLE in copy from "no gallery" (a user should be able to tell "there were photos but we didn't look" from "there were no photos to look at") — do not collapse these into one generic "unavailable" string.
- **Low-confidence / suppressed:** copy per Copywriting Contract ("För osäkert för att visa…"), same treatment. Only shown when the claim list is empty AFTER confidence filtering (i.e. vision ran, found something, but nothing cleared the confidence bar) — distinct from both states above (vision ran, but nothing was assertable).
- **Partial results (some attributes assessable, others not):** each attribute (kitchen/bathroom/overall) that has no assessable claim is simply omitted from the row list — no "not assessed" row is shown for it (mirrors `computeBrfGrade`'s "not_assessable → omit, never fabricate" discipline). The section only shows the empty-state copy above when ALL attributes are non-assessable; if at least one attribute has a valid cited claim, the section renders that claim only, with no filler placeholder for the missing ones.
- **Section-level closing disclaimer:** always rendered at the bottom of `CardContent` whenever at least one claim row is shown — "Kan vara fel — dessa bedömningar är AI:ns tolkning av bilder, inte en besiktning." (`text-xs text-warm-gray-500`, mirrors `BrfScoreCard`'s closing explainer line). Not shown when one of the three empty/degraded states above is the only content (the section-level sub-label under the title already carries the equivalent caveat in that case, avoiding redundant disclaimers stacked on top of an already-empty section).

### 4. "från bildtolkning" marker (cross-cutting, currently dormant)

- **Component:** a small outline `Badge` — `variant="outline"` with `border-terracotta-200 text-terracotta-600` (outline, NOT filled, to visually subordinate it to the vision section's own filled-icon identity) — reading "från bildtolkning".
- **When it must appear:** ANY time a vision-derived signal is displayed adjacent to or inside a ranking/score/flag context outside the vision section's own card (binding constraint — ROADMAP criterion 3 / REQUIREMENTS DISC-04 / Out-of-Scope "vision output feeding the deterministic flag/score system"). Phase 11's own scope does not create such a call site (vision output stays inside its own card, is not fed into Phase 10's `computeNicheScore` or `ReportFlags`), so as of this phase the marker has zero live render locations — it is specified here so Phase 12 or any future integration has an unambiguous, pre-approved treatment rather than inventing one ad hoc.
