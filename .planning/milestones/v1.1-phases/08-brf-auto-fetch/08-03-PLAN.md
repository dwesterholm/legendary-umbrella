<objective>
Build the backend action layer that ties Plan 02's source modules to Plan 01's shared extraction core, gated by the SAME auth/ownership discipline as `analyzeBrf`. Two `"use server"` actions in `src/actions/fetch-brf-auto.ts`:

1. `resolveOrgNrAction(analysisId)` — auth+ownership gate, reads the persisted listing's `brfName` + kommun (from breadcrumbs/address), calls `searchAllabrfByName` → `resolveOrgNr`, and returns a discriminated result the UI uses to decide: `high` → show the confirmation step (org.nr + a fetched fiscal-year preview); `low`/`none` → tell the UI to fall straight through to manual upload. Does NOT fetch/analyze yet.
2. `confirmAndAnalyze(analysisId, orgNr, fiscalYear)` — invoked AFTER the user confirms (ENRICH-02). Auth+ownership gate, writes `brf_status = "auto_fetching"`, calls `fetchArsredovisning(orgNr)`, and on success delegates the extracted iXBRL text to `runBrfExtraction(analysisId, user.id, { kind: "ixbrl-text", text }, "auto_allabrf")` — the IDENTICAL pipeline the manual upload runs. On any fetch failure it releases the status (to a fall-through state) and returns a discriminated "fall through to manual" result — never a wrong-BRF analysis.

Redundant-work guard (RESEARCH Security Domain — cost/abuse): a re-triggered auto-fetch while `brf_status` is already `auto_fetching` (or `done`) must not re-scrape/re-bill — check the current status before starting, mirroring the CAS in-flight-lock idea from `generateReport`.

Purpose: The orchestration seam that makes ENRICH-01 (auto-fetch → identical pipeline) and ENRICH-02 (confirm before analyze; fall through on failure) real, behind the mandatory auth gate.
Output: `src/actions/fetch-brf-auto.ts` + tests.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client → server action (analysisId, orgNr, fiscalYear) | Client-supplied params must be re-validated + ownership-checked server-side |
| Server action → Supabase analyses row | Ownership-gated status/data writes; RLS backstop |
| Server action → Allabrf (via Plan 02 modules) | External fetch trust boundary (SSRF handled in Plan 02) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-10 | Elevation of Privilege | Auto-fetch actions reachable without ownership | mitigate | Both actions open with the copied `supabase.auth.getUser()` + `row.user_id === user.id` gate (analyze-brf 174-193); no resolution/fetch/analysis runs before it passes (ASVS V4). Tests assert the gate blocks before any external call. |
| T-08-11 | Spoofing / Tampering | Client passes a forged/mismatched org.nr to `confirmAndAnalyze` | mitigate | `confirmAndAnalyze` re-validates the org.nr with `isValidOrgNr` server-side (never trusts the client value); the user-confirmation step (Plan 04) is the human-in-the-loop check per ENRICH-02, applied unconditionally even on a high-confidence match. |
| T-08-12 | Denial of Service (cost/abuse) | Repeated auto-fetch triggering redundant scraping + Claude billing | mitigate | Redundant-work guard: early-return when `brf_status` is already `auto_fetching`/`done` (CAS-in-flight idiom from generateReport); the D-06 content-hash cache in runBrfExtraction dedupes identical documents once fetched. |
| T-08-13 | Information Disclosure | Wrong-BRF financials persisted to a user's row | mitigate | The document is only fetched for an org.nr that passed the resolver's high-confidence gate (Plan 02) AND the user confirmed (Plan 04); a fetch failure returns fall-through, never a wrong-BRF write. Ownership check ensures the write targets only the caller's own row. |
| T-08-14 | Information Disclosure | GDPR-safe logging | mitigate | Log only `{ analysisId, orgNr-presence, code }` — never scraped HTML, financials, or the fetched document text. |
| T-08-SC | Tampering | npm/pip/cargo installs | mitigate | No new packages introduced. slopcheck N/A (08-RESEARCH.md Package Legitimacy Audit: zero new packages). |
</threat_model>
