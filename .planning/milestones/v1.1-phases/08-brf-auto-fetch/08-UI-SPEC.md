## Design System

| Property | Value |
|----------|-------|
| Tool | shadcn (already initialized — `components.json` present, detected not asked) |
| Preset | style: `new-york`, baseColor: `neutral`, cssVariables: true, rsc: true (`components.json`) |

This phase's palette, type scale, and spacing are **entirely inherited** from the shipped warm-earthy design language (Phases 2–7) — no new tokens are introduced. All values below are pre-populated from the existing codebase (`globals.css`, `brf-section.tsx`, `brf-progress.tsx`, `brf-score-card.tsx`, `listing-summary.tsx`, `report-flags.tsx`), confirmed as the contract for this phase's new UI.

## Typography

Pre-populated from the existing BRF/listing components — same 4 roles, same 2 weights, reused verbatim (no new sizes introduced):

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px (`text-sm`) | 400 (regular) | 1.5 (Tailwind `leading-relaxed` where prose, default elsewhere) |

| Heading | 18px (`text-lg`) | 500 (medium) | 1.2 |
| Display | 24px (`text-2xl`) | 600 (semibold) | 1.2 |

Weights actually used across the codebase: `font-medium` (500) and `font-semibold` (600) — this phase uses ONLY these two, matching `brf-section.tsx` (`font-medium` headings) and `listing-summary.tsx`/`brf-score-card.tsx` (`font-semibold` for CardTitle/values). Regular (400) is the Tailwind default body weight, not a declared "weight token" — so the 2-weight declaration for emphasis is `medium` + `semibold`.

## Color

Pre-populated from `globals.css` OKLCH tokens + the warm-palette custom colors already governing every BRF/listing component. No new colors introduced for this phase.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `--color-warm-white` / `bg-background` (oklch(0.98 0.005 80)) | Page background, card surfaces |
| Secondary (30%) | `--color-warm-gray-50`/`100` (`bg-warm-gray-50`) + `--card` (white) | Metric tiles, card bodies, dashed-border upload containers |
| Accent (10%) | `--color-sage-600` (oklch(0.50 0.08 150), `bg-sage-600`) | Primary confirm CTA, active progress step, provenance "auto-fetched" success signal |
| Destructive | `--color-destructive` (oklch(0.577 0.245 27.325)) + `--color-terracotta-*` | Reject/manual-fallback affordance emphasis, error banners, staleness/mismatch warnings |

Accent (`sage-600`/`sage-700`) reserved for, explicitly:
- The confirmation step's primary "Ja, stämmer" (confirm) button
- The active/completed step indicators in the `auto_fetching` progress list (mirrors `brf-progress.tsx`'s existing sage dots)
- The "Källa: Bolagsverket" / "Källa: Allabrf" provenance caption text is NOT sage — provenance captions stay `text-warm-gray-500` (neutral), matching the existing `SOURCE_CAPTIONS` convention in `listing-summary.tsx`. Sage is reserved for interactive/success affordances only, never for plain metadata text.

Terracotta (`terracotta-50`/`500`/`600`) reserved for:
- The fiscal-year staleness flag banner ("Detta är inte det senaste tillgängliga räkenskapsåret")
- The auto-fetch-failed / ambiguous-match banner that hands off to manual upload
- The "Nej, det stämmer inte" (reject) button — styled as a neutral/secondary outline button (NOT `variant="destructive"` red), since rejecting is a normal disambiguation path, not a destructive data-loss action. Reject uses `variant="outline"` with default border, matching how `brf-score-card.tsx`'s "Avbryt" cancel action is styled (`variant="ghost"`) rather than red.

All copy in Swedish, matching the established informal-but-precise register ("Analyserar", "Ej tillgänglig", "Källa: X") seen across `brf-progress.tsx` / `listing-summary.tsx` / `report-flags.tsx`. New strings for this phase:

| Element | Copy |
|---------|------|
| Confirmation step heading | "Stämmer detta med din bostad?" (locked — verbatim from ROADMAP/CONTEXT success criteria) |
| Confirmation step body | "Vi hittade en årsredovisning som verkar matcha. Kontrollera att uppgifterna nedan stämmer innan vi analyserar den." |
| Confirmation detail — org.nr | Label: "Organisationsnummer" · Value rendered as-is (e.g. "769600-1234") |
| Confirmation detail — fiscal year | Label: "Räkenskapsår" · Value rendered as the year (e.g. "2024") |
| Confirmation detail — matched BRF name | Label: "Bostadsrättsförening" · Value: `brfName` (if available from Phase 6) |
| Primary CTA (confirm) | "Ja, stämmer — analysera" |
| Secondary CTA (reject) | "Nej, ladda upp manuellt" — routes straight into the existing `BrfUpload` component, no extra dialog |
| Auto-fetch progress heading | "Hämtar årsredovisningen automatiskt…" (mirrors `brf-progress.tsx`'s `"Analyserar årsredovisningen"` heading style/weight) |
| Auto-fetch progress steps | "Söker organisationsnummer…" → "Hämtar dokument…" → "Förbereder analys…" (then hands off into the existing `BrfProgress` reading/extracting/scoring steps once the document is in hand — auto-fetch has its OWN short pre-step list, distinct from and prepended to the existing 3-step list, never merged into one confusing 6-step list) |
| Auto-fetch failed / ambiguous → fallback banner | "Vi kunde inte hitta en säker matchning automatiskt. Ladda upp årsredovisningen manuellt istället." (terracotta banner, identical tone/style to `brf-section.tsx`'s existing failed-view banner) |
| Empty state (no org.nr resolvable, auto-fetch never attempted) | No separate empty state screen — when no org.nr is resolvable, the auto-fetch step is skipped entirely and the user lands directly on the existing manual upload view (`BrfUpload`) with no error banner, since nothing failed — auto-fetch was simply never attempted. This is a silent, honest degrade, not an error. |
| Staleness flag (report) — label | "Räkenskapsår {year}" always shown prominently next to/inside the BRF card heading area |
| Staleness flag (report) — warning copy | "Nyare årsredovisning kan finnas — denna är från {year}, inte det senaste tillgängliga räkenskapsåret." ## Component Inventory & States (for planner/executor)

Scoped strictly to the 4 UI surfaces named in the objective. Reuses existing components; only 2 new components are needed.

### 1. New: `BrfMatchConfirmation` (pre-analysis confirmation step)
- Renders BEFORE `BrfProgress`/auto-fetch begins the actual document fetch — i.e., after a high-confidence org.nr match is found, before extraction runs.
- Layout: `Card` (`w-full max-w-2xl border-warm-gray-200`), `CardHeader` with the heading + body copy, `CardContent` with a 2–3 row key/value detail block (org.nr, räkenskapsår, BRF name if known) styled like `MetricCard` in `listing-summary.tsx` (`rounded-lg bg-warm-gray-50 p-4`, uppercase label + semibold value).
- Actions: two buttons side by side (`flex gap-2` or `gap-3`) — confirm (`bg-sage-600 text-white hover:bg-sage-700 h-11 px-6`) and reject (`variant="outline" h-11 px-6`). Confirm is visually primary (filled), reject is visually secondary (outline) — but BOTH are equally sized/equally reachable; reject is never smaller or ghost-styled, since declining a wrong match is a first-class, expected outcome, not an escape hatch.
- On confirm → transitions to auto-fetch progress state (#2). On reject → transitions directly to the existing manual `BrfUpload` view (#3), no intermediate screen.

### 2. New: `BrfAutoFetchProgress` (transient `auto_fetching` loading state)
- Mirrors `BrfProgress`'s exact visual structure (same `Card`, same step-dot list styling: `bg-sage-600 text-white` done, `bg-sage-100 text-sage-700` + spinning ring active, `bg-warm-gray-100 text-warm-gray-400` pending) — copy only the visual pattern, not the component, since this is a distinct short pre-step sequence (see Copywriting table) that completes BEFORE the existing 3-step `BrfProgress` sequence begins.
- Poll cadence: reuse the same `POLL_MS = 1500` / `MAX_POLL_MS = 90_000` constants and Supabase-row-polling approach (`brf_status = "auto_fetching"` is the new transient status this step polls for, per ROADMAP/CONTEXT's `auto_fetching` status decision).
- On timeout or failure → same `timedOut` treatment as `BrfProgress` ("Det tar längre tid än väntat...") but the terminal failure path routes to the fallback banner + manual upload (per Copywriting table), not a bare retry.

### 3. Existing, unchanged: `BrfUpload` (manual fallback — equally prominent)
- No visual changes required. `BrfSection`'s orchestration is extended with a new initial branch: if a high-confidence org.nr match exists, show `BrfMatchConfirmation` first; otherwise (no match, low-confidence, or after reject) go straight to today's `BrfUpload` view exactly as it renders now — same dashed-border container, same helper copy, same "Analysera BRF" button.
- Side-by-side prominence rule: `BrfMatchConfirmation`'s reject action must lead here with ZERO added friction (no extra "are you sure" step) — reinforcing that manual upload is a first-class path, never a demoted/hidden one.

### 4. Extended: `BrfScoreCard` fiscal-year + provenance header row
- Add a small header row above or beside the existing grade banner: "Räkenskapsår {year}" using the `text-sm text-warm-gray-500` caption style already used for `gradeCaption`, PLUS the new source caption ("Källa: Bolagsverket" / "Källa: Allabrf" / "Källa: Manuellt uppladdad") directly beneath it in `text-xs text-warm-gray-500` (identical styling to `listing-summary.tsx`'s `sourceCaptionFor` captions).
- Staleness flag: when the fetched fiscal year is NOT the most recent available (a boolean/latest-year comparison the backend must supply), render the terracotta staleness caption (see Copywriting table) directly under the fiscal-year label — same `text-terracotta-600` treatment as other warning captions in this component (`scanned` banner precedent), but inline/scoped to the fiscal-year field, not a full top-of-card banner (a full banner is reserved for whole-document issues like `scanned`, not for a single-field caveat).
