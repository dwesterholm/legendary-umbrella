## Color

Identical OKLCH tokens from `src/app/globals.css` — no new colors:

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `--color-warm-white` / `--background` (oklch(0.98 0.005 80)) | Page background |
| Secondary (30%) | `--color-warm-gray-50/100` + white `Card` surfaces | Cards, metric tiles, filter panel background |
| Accent (10%) | `--color-sage-600` (oklch(0.50 0.08 150)) | **Reserved for:** primary submit CTA ("Starta sökning"), active/done progress-step indicators, links/hover states on candidate cards (`group-hover:border-sage-200`), the "done" status badge |
| Destructive / warning | `--color-terracotta-600` (oklch(0.50 0.14 35)) | Kill-switch degraded-mode banner, cap-reached message, failed-job banner, error text — same soft-warning vocabulary as `ListingSummary`'s `partial`/`brokerFetchFailed` banners |

Accent reserved for: the single submit button on the discovery input form, the "queued → running → done" active-step markers in the progress view (mirroring `BrfProgress`'s `bg-sage-600`/`bg-sage-100` step dots), and the hover border on result candidate cards (mirroring `AnalysisCard`'s `group-hover:border-sage-200`). Accent is NEVER used for the killed/failed/cap-reached states — those are terracotta only, never red/destructive (`--destructive` stays reserved for genuinely destructive user actions, of which this phase has none — see Copywriting Contract).

No new semantic color is introduced. Terracotta already carries three roles in this phase (all "calm warning", never alarming): (1) cap-reached honesty message, (2) kill-switch degraded-mode message, (3) failed/killed job status — all rendered with the identical `bg-terracotta-50` banner + `text-terracotta-600` body pattern already used by `ListingSummary` and `BrfProgress`.

## Feature Flag Contract (LOCKED — CONTEXT.md)

The entire discovery entry point is gated behind a server-checked feature flag, OFF by default, until the operator completes final legal sign-off.

- **Implementation:** a server-only env var (e.g. `DISCOVERY_ENABLED`, checked in a server component/action — never `NEXT_PUBLIC_*`, since the flag must not be inspectable/toggleable from the client bundle).
- **Visual contract when OFF:** the discovery entry point (e.g. a "Sök efter drömbostad" nav link/section on the dashboard) is **not rendered at all** — no disabled button, no "coming soon" badge, no grayed-out card. Conditionally return `null` from the server component before any discovery markup mounts. This mirrors the codebase's existing `notFound()` pattern for absent resources — full absence, not a visible-but-inert state.
- **Direct URL access when OFF:** navigating directly to `/discover` or `/discover/[jobId]` while the flag is off resolves the same way `analysis/[id]` resolves a missing row — `notFound()` (Next.js 404), not a "feature disabled" message (avoids leaking that a hidden feature exists).
- **Visual contract when ON:** entry point renders as a normal dashboard section, styled identically to the existing "Ny analys" `UrlInput` section on `dashboard/page.tsx` (same heading treatment: `text-sm font-medium uppercase tracking-wider text-warm-gray-500`).

## Component Inventory (per UI surface)

### 1. Discovery input (`/discover` or a dashboard section)

- **Layout:** single `Card` (`max-w-2xl`, `border-warm-gray-200` — same card shell as `UrlInput`'s form width and `ListingSummary`'s card border), stacked `space-y-4` internally.
- **Free-text field:** shadcn `Textarea`-equivalent (project has no `textarea.tsx` yet — **DISCRETION (resolved):** run `npx shadcn add textarea`, official registry, matches the `Input` styling contract: `border-warm-gray-200 focus-visible:ring-sage-500`). `min-h-24`, placeholder "Beskriv din drömbostad…", label "Vad letar du efter?" above it (`text-xs font-medium uppercase tracking-wider text-warm-gray-500`, matching `MetricCard`'s label treatment).
- **Hard filters row:** a `grid grid-cols-2 gap-4 md:grid-cols-4` row below the textarea (mirrors `ListingSummary`'s `grid-cols-2 md:grid-cols-3` metric grid rhythm): Område (`Select`, options from the Phase 5 owned-client's known area/kommun list), Prisintervall (two `Input type="number"` min/max, or a single range control — DISCRETION: two plain number inputs, lowest-risk, no new range-slider component needed), Antal rum (`Select`, e.g. 1–5+), Storlek kvm (two `Input type="number"` min/max). All filters are optional; only the free-text field is required to submit.
- **Submit button:** `Button` styled exactly like `UrlInput`'s submit — `bg-sage-600 text-white hover:bg-sage-700 h-11 px-6`, label "Starta sökning", spinner + "Startar..." while the create-job action is pending (same `animate-spin` border-pattern as `UrlInput`/`AnalysisCard`).
- **Client-side validation:** free-text required (mirrors `UrlInput`'s inline `text-sm text-terracotta-600` error caption for the URL-format check) — e.g. "Beskriv vad du letar efter för att starta sökningen."

### 2. Progress state (`/discover/[jobId]` while job is non-terminal)

- **Component:** a new `DiscoveryProgress` component, directly modeled on `BrfProgress` — same polling shape (`useEffect` interval against a Supabase row `select` on `discovery_jobs`), same `Card` shell (`w-full max-w-2xl border-warm-gray-200`), same terminal-status → `onComplete` callback contract, same `MAX_POLL_MS` safety-ceiling pattern (adapt the ceiling upward since an area scrape of 20–30 listings will run longer than one Haiku BRF extraction — DISCRETION: e.g. `MAX_POLL_MS = 5 * 60_000`, with the same "tar längre tid än väntat, försök igen" timeout message pattern).
- **Status vocabulary:** queued / running / done / failed / killed (LOCKED set from objective). Map to the same step-dot visual as `BrfProgress`'s `STEPS` list is overkill for a single-number progress — DISCRETION (resolved): render **one live counter line** ("12 av 25 annonser analyserade") instead of a multi-step list, since there's one meaningful axis of progress (count), not discrete named phases. Below the counter, a status `Badge` (reusing `src/components/ui/badge.tsx`) showing the current status word in Swedish: "I kö" / "Analyserar" / "Klar" / "Misslyckades" / "Avbruten".
- **Leave-and-return:** since progress is DB-persisted (`discovery_jobs.status`/`processed_count`), reloading `/discover/[jobId]` at any time re-resolves the current state server-side first (same `initialStatus` prop precedent as `BrfProgress`), so the polling component never starts from a false "queued" flash.
- **Cap-reached honesty banner:** once the job's `processed_count` hits its cap (even mid-run, not just at done), show the terracotta banner inline below the counter: "Vi stannade vid {cap} annonser (sökgräns)." This can appear WHILE status is still "running" (the scrape stopped scanning further but is still scoring what it has) — treat cap-reached as an orthogonal boolean flag on the job row, not a status value, so it can compose with running/done/failed.
- **Failed/killed terminal states:** same `Card` shell, counter frozen at last known value, terracotta banner with the failed/killed copy from the Copywriting Contract above, plus a "Försök igen" link back to `/discover` (not an auto-retry).

### 3. Results (`/discover/[jobId]` once terminal, or a `results` tab within the same route)

- **Layout:** page heading (`text-2xl font-semibold text-warm-gray-900`, same as `DashboardPage`'s "Översikt"), sub-line showing the original free-text query + filters used (so the user remembers what they searched for), then either the empty state or the candidate grid.
- **Candidate grid:** `grid grid-cols-1 gap-4 md:grid-cols-2 lg:grid-cols-3` — byte-identical grid classes to `DashboardPage`'s analysis-card grid.
- **Candidate card:** a new `DiscoveryCandidateCard`, directly modeled on `AnalysisCard` (`Link` wrapping a `Card`, `group-hover:shadow-md group-hover:border-sage-200` hover treatment): address as `CardTitle`, price prominent (`formatSEK`), the same compact 3-column metrics row (Pris/kvm, Rum, Storlek) reusing `formatSEK`/the "---" missing-value convention. Adds one new element beneath the metrics row: the "Källa: Booli" provenance caption (`text-xs text-warm-gray-500`) and a "Se full analys" link/button. No match score, no ranking badge, no star rating (Phase 10 scope, not this phase — see no-prediction/no-verdict constraint above).
- **Empty state:** same visual pattern as `DashboardPage`'s "Inga analyser än" block — dashed-border `warm-gray-200` box, `warm-gray-50` fill, centered icon (reuse a Lucide `SearchX` or similar rather than the inline SVG `DashboardPage` currently hand-rolls — DISCRETION: prefer `lucide-react` icon import for consistency with the rest of the codebase's icon usage, `h-12 w-12 text-warm-gray-300`), heading "Inga träffar denna gång", body copy from the Copywriting Contract.
- **Failed/killed results view:** if the user lands on `/discover/[jobId]` after the job already failed/was killed, skip the progress component entirely and render the terminal banner + retry link directly under the page heading (no flash of a progress bar for an already-dead job).

### 4. Cost/limit honesty + kill-switch messaging (cross-cutting)

- **Visual pattern:** identical terracotta banner treatment used three ways (cap-reached, kill-switch degraded, failed) — `rounded-t-xl bg-terracotta-50 px-6 py-3` when it's the top of a `Card` (matches `ListingSummary`'s existing banner), or `rounded-lg bg-terracotta-50 px-4 py-3` as a standalone inline block elsewhere. Always `text-sm text-terracotta-600` body text. Never a `destructive`-red alert — this is calm/informational, not alarming, matching the existing "soft warning" register used for `partial`/`brokerFetchFailed`.
- **Kill-switch entry point message:** if the kill switch has tripped globally (Booli blocking detected), the discovery input form itself (surface #1) should show this banner ABOVE the form and disable the submit button — same "still visible, calmly explained, with an escape hatch" pattern as the degraded-mode message, never a silent full removal (that treatment is reserved for the feature-flag-off case, which is a deliberate product decision, not a live incident).
- **Never exceeds cap:** the UI never implies more work is happening than the cap allows — the counter's denominator is always `min(scanned, cap)`, so "25 av 25" reads as complete-at-cap, not as "there were exactly 25 listings."
