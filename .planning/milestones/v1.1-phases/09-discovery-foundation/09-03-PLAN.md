Logic in order: (1) INCREMENTAL CAP GATE — if `claimedRow.candidate_count >= claimedRow.cap_candidates` OR `claimedRow.cost_sek_total >= claimedRow.cap_sek`, UPDATE the row `status='done', cap_reached=true` and return (do NOT scrape). (2) Resolve areaId: `resolveArea(filters.areaQuery)`; if null, flip `status='failed'` and return. (3) COST PRE-CHECK — if `claimedRow.cost_sek_total + estimatedSliceCost > claimedRow.cap_sek`, flip `status='done', cap_reached=true` and return. (4) KILL SWITCH + scrape — `try { const raw = await fetchAreaListings(areaId, filters.objectType) } catch (killSignal) { UPDATE status='degraded'; log [discovery] kill-switch degraded; return }`. (5) Map: `raw.map(toCandidate)` → `filterCandidates(candidates, filters, cap_candidates - candidate_count)` → `{ shown, scanned }`. (6) PERSIST — single UPDATE writing `results = existing ++ shown` (PII-safe candidates only), `candidate_count += shown.length`, `processed_count += scanned`, `cost_sek_total += discoveryCostSek({haikuUsage:0-usage, renders})`, and `status='done'` + `cap_reached` if the new candidate_count hits cap_candidates (else leave 'processing' for the next tick). All counters are computed from `claimedRow`'s returned values, never a fresh SELECT.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → startDiscovery/tickDiscovery | Untrusted FormData + jobId cross into server actions; the feature flag, auth, ownership, and caps are all enforced server-side. |
| free text → Haiku call | Untrusted natural language crosses into an LLM prompt — sent as user-message data, never as system-prompt instruction. |
| owned Booli client → job | A blocking/CAPTCHA signal (a thrown dead-source error) crosses into the orchestrator, which must degrade-and-halt. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-07 | Tampering (prompt injection) | `parse-intent.ts` | mitigate | Free text is `messages:[{role:"user",content:freeText}]` only, never merged into `system`; output is a rigid `zodOutputFormat`-validated `DiscoveryFilter`, and the deterministic in-code `filterCandidates` runs AFTER the parse — a successful injection can at most produce a valid filter object (09-RESEARCH.md prompt-injection row). |
| T-09-08 | Elevation of Privilege (flag bypass) | `startDiscovery` | mitigate | `process.env.DISCOVERY_ENABLED !== "true"` is the literal first line, before auth/parse/insert (Pitfall 5) — direct curl/devtools calls fail closed even with the UI hidden. Tested via the flag-off short-circuit assertion. |
| T-09-09 | Denial of Service (cost) | `startDiscovery` + `runSlice` | mitigate | Per-user-per-day job cap in `startDiscovery` (rapid-job-creation bypass) PLUS per-slice incremental candidate/SEK caps read from the claimed row before each scrape (`runSlice`) — never checked only at job end. |
| T-09-10 | Denial of Service (kill switch) | `runSlice` + `fetchAreaListings` | mitigate | A blocking/CAPTCHA signal (owned-client throw) → `status='degraded'` + halt; no retry-into-spend; the UI (Plan 04) surfaces the single-URL escape hatch. |
| T-09-06 | Information Disclosure (IDOR) | `tickDiscovery` | mitigate | Ownership pre-check (`row.user_id !== user.id`) before the RPC claim, behind RLS — a user cannot drive another user's job. |
| T-09-05 | Information Disclosure (PII) | `runSlice` persist | mitigate | Only `toCandidate` allowlist shapes appended to `results`; raw Apollo/broker text never persisted (asserted in job.test.ts). |
| T-09-SC | Tampering | package installs | accept | No new packages this plan. |
</threat_model>
