<objective>
Resolve the one genuinely-new problem RESEARCH flagged as BLOCKING-within-phase: turning a free-text Swedish area name (e.g. "Södermalm", "Nacka") into Booli's opaque `areaId`. Booli has NO documented free-text area endpoint and the existing `resolveAreaId` only works from an existing listing's breadcrumbs. This plan (a) runs the operator-approved Wave-0 Playwright probe that types into Booli's own search box and captures the resulting `areaIds=` value, AND (b) ships a static human-curated seed-list fallback so the phase does NOT hard-block if the live probe proves unreliable. The rest of discovery (Plan 03's scrape) works against either path via the same `{ areaId }` return.

Purpose: Unblock the scrape path without hardcoding a stale comprehensive lookup table (RESEARCH Pitfall 3) and without hard-depending on a probe RESEARCH rated MEDIUM-confidence/UNVERIFIED-live.
Output: `src/lib/booli/area-search-page-function.ts`, `src/lib/discovery/resolve-area.ts` + test, `src/lib/discovery/area-seed.ts`.
</objective>
