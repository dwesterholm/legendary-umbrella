# Phase 12 вҖ” UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Scope note:** This phase introduces NO new visual language. It is a content extension of Phase 11's "AI-bedГ¶mning av bilder вҖ” kan vara fel" vision section, plus ONE new sub-treatment for a signal that is NOT vision-derived (sun-path is `suncalc` math, not an image interpretation). Two distinct content types land this phase:

1. **Floor-plan investigation-prompts** (DISC-05) вҖ” vision-derived, image-cited, hedged. Rendered as an ADDITIONAL claim type inside the SAME `GalleryConditionVision` card Phase 11 built (same shell, same Eye/terracotta identity, same citation row pattern) вҖ” never a new sibling card, never a new visual identity.
2. **Sun-path (theoretical)** (DISC-06) вҖ” deterministic, computed in code from `latitude`/`longitude`/`floor`/orientation via `suncalc`. This is explicitly NOT an AI interpretation of an image, so it must NOT borrow the `Eye`/terracotta "image-interpreted" identity wholesale. It gets its own compact sub-block, still living inside the same vision-section CARD (both are "advisory, not verdict" вҖ” CONTEXT.md's structural-separation constraint is about deterministic FLAGS vs. hedged/advisory content, and sun-path is advisory even though it's computed), but visually marked as "computed, not interpreted" via a distinct icon (`Compass`, not `Eye`) and a neutral (not terracotta) surface treatment.

Both content types are behind `DISCOVERY_ENABLED` (Phase 9's Feature Flag Contract applies unchanged вҖ” full absence when off, never a disabled/greyed-out state). No verdict framing anywhere in this phase.

**Accent reserved for (explicit list, binding, extends Phase 11's list):**
1. Sage (`sage-600`/`sage-100`) вҖ” deterministic-flag "green" severity, rank badges, selected niche. NEVER appears in floor-plan or sun-path content.
2. Terracotta (`terracotta-50/100/600`) вҖ” (a) deterministic soft-warning banners (staleness, degenerate ranking, broker-fetch-failed вҖ” unchanged), (b) the vision section's own identity marker (Eye icon badge, section header), AND (c) floor-plan investigation-prompt rows (since floor-plan is vision-derived, it correctly inherits (b)). Sun-path does NOT use terracotta вҖ” see the `Compass`/neutral-gray row above.
3. Destructive (`--color-destructive`) вҖ” deterministic red-severity flags only, and the "OsГӨker вҖ” kontrollera sjГӨlv" BRF badge. NEVER used for floor-plan or sun-path content, no matter how low-confidence or unavailable вҖ” unavailability is communicated via copy ("ej tillgГӨnglig"), never via red/destructive styling.
4. Warm-gray-500 on warm-gray-100 (NEW pairing this phase, existing tokens) вҖ” reserved EXCLUSIVELY for the sun-path `Compass` icon badge and its sub-block surface. Do not reuse this exact pairing (gray icon badge + gray-50 sub-block) for any future "computed, not interpreted" content type without checking this precedent first вҖ” it is now the codebase's vocabulary for "deterministic math shown inside an otherwise-hedged/advisory section."

**Why floor-plan reuses terracotta but sun-path does not:** Floor-plan claims are an LLM's interpretation of a 2D image вҖ” exactly the same epistemic category as "kГ¶ket verkar renoverat." They belong in the identical citation-row shell with the identical Eye/terracotta identity; inventing a different treatment for floor-plan specifically would fragment the "image-interpreted" vocabulary Phase 11 established for no reason. Sun-path is the opposite: it is exact astronomical trigonometry over `latitude`/`longitude`/date вҖ” zero LLM involvement, zero interpretation, zero chance of hallucination. Presenting it identically to the vision claims would falsely imply the same evidentiary weakness (or false strength) as an AI's guess about a kitchen photo. The distinction must be visible even at a skim: same card, different icon (`Eye` vs `Compass`), different sub-block surface tone (`bg-warm-white` claim rows vs `bg-warm-gray-50` sun-path grid).

## Component Inventory (per UI surface)

### 1. Floor-plan investigation-prompt rows вҖ” extends existing `gallery-condition-vision.tsx`

- **No new card, no new component file for the row shell.** Floor-plan claims are a FOURTH `attribute` value (alongside `kitchen`/`bathroom`/`overall`) on the SAME `VisionConditionClaim`/`VisionResult` shape and render through the EXACT SAME claim-row markup Phase 11 already built (`flex items-start gap-3 rounded-lg border border-warm-gray-100 bg-warm-white p-3`, thumbnail + "Bild {n}" citation, attribute label, hedged claim text) вҖ” only `ATTRIBUTE_LABELS` gains a `floorPlan: "PLANLГ–SNING"` entry.
- **Citation target:** the cited image for a floor-plan claim MUST be the floor-plan image itself (already extracted first/prioritized by `extractImageUrls` in `client.ts` вҖ” floor plans are ordered before gallery photos in `imageUrls`). The thumbnail crop shows the floor-plan image, not a gallery photo, for any claim with `attribute === "floorPlan"`.
- **Mandatory disclaimer placement:** unlike kitchen/bathroom/overall (which share ONE section-level closing disclaimer), each INDIVIDUAL floor-plan claim's `claim` text itself must embed or be immediately followed by "krГӨver konstruktГ¶r / vГӨggutredning" вҖ” this is stronger than the section-level disclaimer because a floor-plan claim carries materially higher misreading risk (a user could act on a wall-removal idea) than a "kitchen looks renovated" claim. Implementation: the claim string generated by the vision pass includes the phrase verbatim (schema/prompt concern, not a UI concern вҖ” but the UI must never truncate or hide it, e.g. no `line-clamp` on floor-plan claim text specifically).
- **Ordering within the claim list:** floor-plan claims render in the SAME flat list as kitchen/bathroom/overall (no visual grouping/sub-header needed) вҖ” Phase 11's existing `claims.map(...)` loop needs no structural change, only the new attribute label + the citation-to-floor-plan-image behavior above.

### 2. Sun-path sub-block (new, lives INSIDE `gallery-condition-vision.tsx`'s `CardContent`, below the claim list)

- **Structural position:** a distinct sub-block rendered AFTER the vision-claims list (floor-plan + kitchen/bathroom/overall) and AFTER the Phase 11 closing disclaimer, still inside the SAME outer `Card` вҖ” no new outer card, no new page-level section, no additional `space-y-6` gap needed (this is content within the existing vision section, not a new section). Separated from the claims list above it by a `pt-4 border-t border-warm-gray-100` divider (`md`/16px top padding above the border, within the declared {4,8,16,24,вҖҰ} spacing set) so it reads as "a related but distinct sub-topic," not a continuation of the claim list.
- **Sub-header:** `flex items-center gap-2` row with a `Compass` icon badge (`flex h-6 w-6 items-center justify-center rounded-full bg-warm-gray-100`, `<Compass className="h-3.5 w-3.5 text-warm-gray-500" />` вҖ” smaller (`h-6 w-6`/`h-3.5 w-3.5`) than the vision section's own `h-8 w-8`/`h-4 w-4` Eye badge, since this is a sub-heading within the card, not the card's primary identity) beside the "Solexponering" `CardTitle`-weight text (`text-base font-semibold text-warm-gray-900`), then the theoretical-label sub-text directly below (`text-sm italic text-warm-gray-500`).
- **Grid/table (compact, no charting library):** a plain CSS grid or `<table>`-like structure вҖ” orientation as rows, season as the 3 columns (Vinter / VГҘr-HГ¶st / Sommar), OR orientation as columns and season as rows, whichever reads more naturally for the number of orientations resolved (typically 1вҖ“2 known facades per listing, not all 4 вҖ” render only the orientation(s) actually known, never fabricate the other 3 as "ej tillgГӨnglig" rows). Cell shell: `rounded-md bg-warm-gray-50 border border-warm-gray-100 p-2 text-center`, header cells `text-xs font-medium uppercase tracking-wider text-warm-gray-500`, data cells `text-sm text-warm-gray-700`. No color-coding, no heat-map gradient, no icons-per-cell (CONTEXT.md "no heavy new charting" вҖ” this is intentionally as plain as `BrfScoreCard`'s metric rows, just gridded instead of stacked).
- **Degraded state:** when orientation is unavailable/low-confidence (the expected common case today, per RESEARCH), the grid does not render at all вҖ” replaced by the single degraded-state line (Copywriting Contract) in the same `italic text-warm-gray-500 text-sm` treatment, INSIDE the same sub-header shell (icon + title + sub-label always present, mirrors Phase 11's "section identity always visible even when empty" discipline). Floor is a SEPARATE precondition from orientation вҖ” if floor is known but orientation is not (or vice versa), the result is still "ej tillgГӨnglig" (both are required inputs to a meaningful facade/floor answer); do not partially render a floor-only or orientation-only grid.
- **What is explicitly NOT reused from the vision-claim rows:** no thumbnail, no "Bild {n}" citation (sun-path cites no image вҖ” it is computed from stored numeric fields, not derived from a photo), no `Eye` icon, no terracotta tint anywhere in this sub-block.
- **What is explicitly NOT introduced:** no charting library (`recharts`/`d3`/etc.), no SVG sun-path diagram, no interactive/draggable element (SUN-02 in REQUIREMENTS.md's Future Requirements is explicitly deferred вҖ” this phase is a static table only).

### 3. Degraded / empty states (extends Phase 11's existing state machine)

Both floor-plan and sun-path degrade INDEPENDENTLY of each other and of the kitchen/bathroom/overall claims вҖ” a candidate can have any combination of: gallery claims present + floor-plan present + sun-path present; gallery claims present + floor-plan absent + sun-path unavailable; vision skipped entirely (existing Phase 11 `no_images`/`cost_cap`/`vision_error` states cover the WHOLE vision pass including floor-plan, since floor-plan claims come from the same vision call) + sun-path still independently available (sun-path is NOT a vision output, so `visionSkippedReason` does not gate it).

- **`visionSkippedReason` states (no_images / cost_cap / vision_error):** unchanged from Phase 11 вҖ” these suppress the ENTIRE claims list including floor-plan (floor-plan is vision-derived, so it shares vision's failure modes). Sun-path renders independently below/instead, per its own orientation/floor availability check.
- **No floor-plan image specifically (but gallery photos exist):** the vision pass still ran and may have produced kitchen/bathroom/overall claims; simply no `attribute: "floorPlan"` claim exists in the result (empty-per-attribute omission, same as any other unassessable attribute) вҖ” no special-cased banner needed beyond the existing per-attribute omission rule, UNLESS the floor-plan image was never available AT ALL (not even attempted), in which case the floor-plan-specific empty copy (Copywriting Contract) may optionally render as a single muted line so a user understands there was no floor plan to analyze вҖ” Claude's discretion on whether this is worth a dedicated line vs. silent omission; if implemented, same `italic text-warm-gray-500 text-sm` treatment, placed where the PLANLГ–SNING row would have been.
- **Sun-path unavailable:** independent of all vision states, per Component Inventory В§2 above.
- **Section-level closing disclaimer:** Phase 11's existing "Kan vara fel вҖ” dessa bedГ¶mningar ГӨr AI:ns tolkning av bilder, inte en besiktning." now implicitly covers floor-plan too (floor-plan is a claim like any other); the ADDITIONAL floor-plan-specific reinforcement line (Copywriting Contract) renders only when floor-plan claims exist, directly below them, distinct from and in addition to the general closing disclaimer (which still renders once at the very bottom of `CardContent` as before).

### 4. "frГҘn bildtolkning" marker (cross-cutting, Phase 11 В§4, dormant вҖ” still dormant this phase)

- Unchanged from Phase 11. Neither floor-plan nor sun-path content is fed into `computeNicheScore`/`ReportFlags` this phase either (binding constraint, DISC-05/06 + REQUIREMENTS Out-of-Scope), so this marker still has zero live render locations. Sun-path, being deterministic rather than vision-derived, would ADDITIONALLY need a different marker ("berГӨknat" not "frГҘn bildtolkning") if it were ever surfaced outside this card in the future вҖ” noted here for the same forward-specification reason Phase 11 noted it, not implemented.
