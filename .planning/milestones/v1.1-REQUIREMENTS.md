# Requirements — Milestone v1.1

**Milestone:** v1.1 Owned Data Layer & Intelligent Discovery
**Defined:** 2026-07-06
**Core Value:** Give Swedish home buyers an independent, data-driven analysis of any listing — the one thing their mäklare won't provide.

> Continues from v1.0 (shipped, archived at `.planning/milestones/v1.0-REQUIREMENTS.md`). v1.0 IDs LSTG-01/02, BRF-01/02/03, PRICE-01, AREA-01, RPRT-01/02/03 are validated; this milestone extends LSTG and ENRICH and adds ACQ, MACRO, DISC.

## v1.1 Requirements

### Owned Acquisition Layer (999.6)

- [x] **ACQ-01**: System retrieves a single active listing by its Booli URL through an owned acquisition client that generalizes the proven Playwright/`__APOLLO_STATE__` transport, with a fallback tree (direct → Apify SE-residential proxy → paid actor as last resort)
- [x] **ACQ-02**: System retrieves multiple active listings for an area (searchForSale-equivalent) through the same owned client — the acquisition foundation discovery (DISC) builds on
- [x] **ACQ-03**: Acquisition degrades gracefully and observably — on transport failure it walks down the fallback tree and surfaces source/health, with no silent dependency on the paid actor

### Deeper Listing Extraction (999.2)

- [x] **LSTG-03**: System recovers listing fields Booli lacks (floor/våning, balcony, BRF name, renovation status, full description) by following through to the broker's own page (`agencyListingUrl`) — 06-01 landed floor/balcony/BRF-name from the Apollo entity (no broker fetch needed for these 3); 06-02 landed the SSRF-guarded fetch + JSON-LD-first/DOM-fallback parser for renovation status + description; Plan 03 wires it into `analyzeUrl`
- [x] **LSTG-04**: Broker-sourced fields fill gaps only — never overwrite Booli data — and preserve provenance; a failed broker fetch never fails the primary listing analysis (graceful degradation) — 06-01 landed the `Sourced<T>`/`ListingSource` provenance contract; 06-02 landed `fillGap`/`mergeListingFields` (never-overwrite proven by test) + `fetchBrokerListingPage`'s null-never-throw degradation; Plan 03 wires the live call site into `analyzeUrl`

### Macro Price Context (999.3)

- [x] **MACRO-01**: Market-context layer includes current macro indicators (Riksbank policy rate, inflation, regional price-index trend) sourced from Riksbank SWEA + SCB
- [x] **MACRO-02**: Macro context is strictly descriptive — presented in its own clearly-labeled section, never framed as a price prediction, future-value forecast, or buy/sell verdict

### BRF Auto-Fetch (999 enrichment)

- [x] **ENRICH-01**: System auto-fetches a BRF's årsredovisning from Bolagsverket and Allabrf, matched by organisationsnummer (not name-string)
- [x] **ENRICH-02**: Auto-fetched documents surface their org.nr + fiscal year for user confirmation before analysis; manual PDF upload (BRF-03) remains the dependable fallback when auto-fetch fails or is ambiguous

### AI Free-Text Discovery + Vision (999.7)

- [x] **DISC-01**: User describes desired properties in free text (+ a few hard filters); system translates that into an area search and returns matching candidate listings — 09-01 landed the structured-filter contract (`intentFilterSchema`) + the deterministic in-code `filterCandidates`; 09-03 landed the live `parseIntent` Haiku call + `startDiscovery`/`tickDiscovery` job orchestration; Plan 04 wires the UI
- [x] **DISC-02**: Discovery runs as a bounded background job with hard per-search caps (candidate count, images/listing, total SEK) and progress polling; results persist and are viewable — 09-01 landed the live `discovery_jobs` table + `claim_discovery_slice` atomic RPC + hard-cap constants + `discoveryCostSek`; 09-03 landed the tick-driven `runSlice` job loop with incremental per-slice cap enforcement + the RPC concurrency integration test; Plan 04 wires UI polling
- [x] **DISC-03**: Candidates are ranked against the free-text intent via configurable niches (renovation-upside, turnkey, imminent-stambyte-where-BRF-pays)
- [x] **DISC-04**: System derives soft condition attributes (e.g. kitchen/bathroom dated vs modern) from description + gallery images via vision — each claim cited to its source image, presented as hedged evidence, structurally separate from deterministic flags — 11-01 landed the foundation contracts (`imageUrls` extractor, additive-nullable `DiscoveryCandidate` extension, slim vision citation schema, `CAP_VISION_SEK_MAX`); 11-02 landed the live two-pass Haiku-pre-filter/Sonnet-deep-pass pipeline, the incremental vision cost cap, PII-ignore prompts, and the RUN_LLM_EVALS-gated eval harness; 11-03 landed the structural-separation UI (`GalleryConditionVision`, Eye/terracotta identity, distinct degraded states) + the structural-separation invariant test — code is complete; the operator-deferred live-render check + 20-30-listing validation gate (kill criterion) has not yet run before production sign-off
- [x] **DISC-05**: System analyzes the floor-plan (planlösning) image for remodel potential as an investigation-prompt with an explicit "kräver konstruktör / väggutredning" disclaimer — never a load-bearing or wall-removal verdict
- [x] **DISC-06**: Each listing shows theoretical sun exposure by facade orientation / floor / season (suncalc), explicitly labeled as unobstructed/theoretical (no shadowing modeling)
- [x] **DISC-07**: Area-wide discovery respects a documented legal go/no-go gate — bounded per-search/per-day scrape caps, human-pace rate-limiting, a kill switch, and vision prompts that ignore people/personal documents in photos (GDPR/PII) — 09-01 landed the PII-safe `DiscoveryCandidate` allowlist (`toCandidate`, no-spread, exact-key-tested) + owner-only RLS on `discovery_jobs`; 09-03 landed the flag-first fail-closed `startDiscovery` gate, the per-day job cap, and the `runSlice` kill switch; the legal go/no-go gate DECISION itself remains a separate operator action (STATE.md Blockers), not a Plan 04 code task

## Out of Scope (explicit exclusions)

- **Payment / Stripe (PAY-01, PAY-02)** — deliberately deferred again; this milestone deepens the product before monetizing. Revisit next milestone against the "paying users" kill criteria.
- **Definitive structural / wall-removal verdicts** — liability + fabrication risk; load-bearing status is not determinable from a 2D floor plan (even pros need a formal väggutredning). DISC-05 hedges as investigation-prompts only.
- **Photorealistic "after renovation" visuals** — too easily mistaken for real; violates the no-fabrication principle.
- **Unbounded / continuous area crawling** — capped, user-triggered discovery only (DISC-02/07); continuous crawling changes the legal posture and the product.
- **Obstruction-accurate sun modeling** — requires 3D building geometry the project has no data source for; DISC-06 stays theoretical.
- **Vitec Express Connect (B2B partner API)** — partnership approval + budget dead-end for a solo dev; broker extraction (LSTG-03) uses public-page parsing instead.
- **Vision output feeding the deterministic flag/score system** — architecture + trust violation; vision stays in its own labeled, hedged section (DISC-04).
