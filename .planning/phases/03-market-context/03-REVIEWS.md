# Cross-AI Plan Review — Phase 3

> Reviewer: in-session adversarial pass (fresh context, Claude). NOT an independent cross-model review — no external CLI was available. Treat as a structured self-critique, weaker than a true different-model review.

## Adversarial Review

1. **Summary**

The plan set is unusually well-structured: it correctly identifies the one genuine unknown (sold-price acquisition), gates it behind a Wave-0 spike, mirrors the proven Phase 2 deterministic-core + RLS + RSC patterns, and the AREA-01 path is genuinely de-risked (SCB verified live). However, the plans contain several real gaps that will bite during execution. The most serious is a **guest/anonymous-analysis blind spot**: the entire Phase 1 product flow allows an unauthenticated guest to run one analysis that is NEVER persisted to the DB (no row, no id), yet every Phase 3 plan assumes an `analysisId` row exists with stored `listing_data` — Plan 06's `isGuest` handling and Plan 05's hard auth gate together mean a guest literally cannot see *any* market panel, which is fine, but the plans never state this and Plan 06's "do not add a login wall the CONTEXT did not specify" instruction directly conflicts with Plan 05's hard auth gate. Second, the **spike-failure degradation is not honest end-to-end**: if the spike fails, Plan 04 builds against a *mock* fixture, and nothing in Plans 05/06 prevents shipping a price panel wired to a source that returns nothing — the "honest marker" only covers thin data, not a structurally-dead source. Third, there are **dependency-ordering and same-wave collision risks** (Plan 03 and Plan 04 both Wave 2 depending only on 03-01, but Plan 04 Task 1 needs the `SoldComp` type that Plan 04 itself defines in Task 2, and the deso.geojson artifact is a single-point build-time failure with no fallback verified). Fourth, **`prisPerKvm` can legitimately be `0`** from Phase 1 (the actor fallback), which makes the ±% headline divide-by-zero-adjacent and produces nonsense comparisons that no plan guards against.

2. **Strengths** (genuine)

- AREA-01 is genuinely de-risked: SCB PxWebApi config, rate limits, and the population table were live-tested (03-RESEARCH Sources/Primary), and the "use the actor's lat/lng instead of a geocoder" insight removes a whole external dependency and failure mode.
- The deterministic-core discipline (D-08) is correctly enforced: 03-04 Task 1 explicitly forbids a valuation/verdict field and keeps ±%/trend/confidence as pure arithmetic, with a grep gate. `@anthropic-ai/sdk` is correctly marked NOT-used this phase (03-RESEARCH Standard Stack).
- The schema-push BLOCKING gate (03-02 Task 2) correctly reuses the Phase 2 Plan 02 pattern and the migration correctly avoids re-creating the `002_brf.sql` UPDATE policy (verified: 002 owns `"Users can update own analyses"`; 003 grep-gates against any `create policy`).
- Source-isolation behind `sold-source.ts` (03-04) is the right architecture — it lets the spike outcome (direct GraphQL vs Apify proxy vs paid actor) change without touching callers, and keeps the secret/ToS posture in one file.
- RED-first test scaffolding (03-01 Task 2) with committed offline fixtures and a no-live-call assertion mirrors the Phase 2 Wave-0 precedent correctly.

3. **Concerns**

- **[HIGH] Guest-flow blind spot — Phase 3 silently excludes the entire anonymous funnel, and Plan 06 contradicts Plan 05 on auth.** Verified in `src/actions/analyze.ts:96-124`: an unauthenticated guest's analysis is returned directly and **never inserted into `analyses`** (no row, no id). The analysis detail page `src/app/(app)/analysis/[id]/page.tsx` only renders for a persisted row (RLS + `notFound()`), so a guest never reaches a market panel at all. Yet 03-06 Task 2 instructs "render the panels for any viewer who can read the analysis… do NOT add a new login wall the CONTEXT did not specify," while 03-05 enforces a HARD auth gate on `enrichMarketContext` (Task 1 step 1, T-03-15). These two are in tension: the section will render an enrich trigger to a logged-in owner only, but the `isGuest` prop wiring implies a guest viewing path that cannot exist. The plans never state that Phase 3 is owner-only and inherits the BRF gating reality. Risk: wasted effort on a guest read-only state that's unreachable, plus reviewer confusion about whether Success Criterion 3 must hold for guests.

- **[HIGH] `prisPerKvm` can be `0` (or a stale list-price/m²), poisoning the ±% headline.** Verified in `analyze.ts:73-77` + `listing.ts:42`: `prisPerKvm` falls back to `0` when price or livingArea is missing, and is otherwise the actor's *list* sqm price. 03-04 `computePriceComparison` takes `listingPrisPerKvm` as a plain `number` and computes `deltaPct = ((listing - areaAvg)/areaAvg)*100` — but no plan guards the case where `listingPrisPerKvm === 0` (older/partial listings) or where it is null-equivalent. A `0` listing pris/kvm yields a "-100 % under områdets snitt" headline that is confidently wrong and undermines the whole trust model (D-09). The compare RED tests (03-01 Task 2) only test the happy `95200` case and thin-comps; there is no `listingPrisPerKvm <= 0` case.

- **[HIGH] Spike-failure path is not honestly degraded through Plans 04→06.** 03-01 Task 1 allows recording a blocker + a **mock** sold-comps fixture so Plan 04 can build. But nothing downstream gates PRICE-01 *display* on a *live* source actually working. 03-04 builds `fetchSoldComps` against the recipe; 03-05 wraps it in try/catch and sets `price = null` on failure; 03-06 then shows "Prisjämförelse ej tillgänglig — för få försäljningar." That message is a **lie** when the truth is "we have no working sold-price source at all," not "this area is thin." The phase can pass all automated gates (types compile, tests green on mock fixtures, build OK, area panel renders) and ship a permanently-empty price panel mislabeled as thin data. ROADMAP Success Criterion 1 (price comparison visible) would be unmet while the phase reports "done." There is no `market_status: "source_unavailable"` distinct from thin/failed, and no checkpoint that blocks phase completion if the live sold source never returned real data.

- **[MEDIUM] `deso.geojson` build-time artifact is a single point of failure with an unverified toolchain and no confirmed fallback.** 03-03 Task 1 requires converting SCB's DeSO GeoPackage to GeoJSON, but 03-RESEARCH Environment Availability marks `ogr2ogr`/GDAL as "✗ not checked on dev machine" and the gpkg format/size as `[ASSUMED]`. The fallback ("kommun.geojson if DeSO too large, or an online converter") is hand-waved. A full DeSO GeoJSON for all ~6,000 DeSO polygons at full vertex resolution can be tens of MB — committing that to the repo and importing it into an RSC/server bundle has real bundle-size and cold-start cost the plans never quantify. If conversion fails, AREA-01 silently collapses to kommun-only, degrading the D-06 "upgrade to neighborhood" promise without any plan acknowledging it as a likely outcome rather than an edge case.

- **[MEDIUM] Intra-plan task ordering hazard in 03-04: `SoldComp` type used in Task 1 but defined in Task 2.** 03-04 Task 1 (compare.ts) imports `SoldComp` from sold-schema.ts, but sold-schema.ts is written in Task 2. The plan hand-waves this ("If Task 2 has not run yet… define the `SoldComp` type in sold-schema.ts first"). With `autonomous: true` and tasks executed in order, Task 1's `npx tsc`/vitest verify will fail on the missing module unless the executor pre-creates the file. This is a self-inflicted RED that the verify gate (`tsc … || echo TYPES-OK`) may mask. Tasks should be reordered (schema first) or the type co-located.

- **[MEDIUM] SCB DeSO income/tenure availability is `[ASSUMED]` (A3), but Plans 03/05/06 treat DeSO-level four-metric data as the normal path.** 03-RESEARCH A3 is MEDIUM: only the *population* table was verified live at DeSO; income (`HE0110I/Tab*InkDesoRegso`) and tenure (`HE0111YDeSo`) existence-at-DeSO-for-a-recent-year is unconfirmed. Pitfall 3 acknowledges this, and 03-03 Task 3 says "fall back to kommun or omit." But there is no Wave-0 spike step that actually queries those two tables at DeSO (unlike the sold-source, which got a whole spike). The realistic outcome — income/tenure only available at kommun — means the AREA-01 "neighborhood demographics" headline (AREA-01 names *income levels*) may in practice always be kommun-level, weakening the requirement. This deserves a live check in 03-01 or 03-03 Task 1, not a runtime fallback discovered in production.

- **[MEDIUM] No per-user rate limit on `enrichMarketContext` — cost-bomb is only partially mitigated, and the sold-source cost is harder to bound than BRF's.** 03-05 T-03-17 explicitly defers per-user aggregate rate-limiting (same honest-scope note as BRF CR-02). But the BRF action's per-call spend is inherently bounded by a single `max_tokens` Haiku call; the sold-source path walks a **tiered fallback (building → neighborhood → wide)** which can mean *multiple* Apify-proxy GraphQL calls per analysis, plus `searchLocations` for areaId resolution. The `soldSourceCostSek` cap gates *persistence* after the fact (like BRF), not the spend. A logged-in user (or a script with one session) re-triggering enrichment repeatedly bills each multi-call walk in full against the <$100/month budget. The cost-cap-on-persist is weaker here than the plan implies because the expensive part (the fetch walk) already happened before the cap is checked.

- **[MEDIUM] SCB module-level in-memory cache does not survive serverless cold starts — the rate-limit mitigation may be illusory.** 03-03 Task 3 proposes "a module-level in-memory cache (or a simple persisted cache key)." On Vercel/serverless (the likely host given <$100/mo + Next.js), module-level memory is per-instance and evaporates between invocations, so "a warm analysis is zero SCB calls" is optimistic. Under any real concurrency the 30-calls/10s limit (Pitfall 4) is reachable. The plan offers "or a simple persisted cache key" as an aside but never commits to it; without a persisted cache (e.g. an SCB-results table or the `area_data` column itself acting as cache), the rate-limit guard is weak.

- **[LOW] `breadcrumbs` shape is `unknown[] | null` and is the areaId join key, but its structure is never pinned before Plan 04 depends on it.** 03-01 Task 3 models breadcrumbs permissively as `unknown[]`. 03-04's tiered fallback (D-01) resolves areaId from "breadcrumbs/booliId." If the spike (03-01 Task 1) doesn't capture the breadcrumbs structure precisely, Plan 04's tier walk is built on an untyped blob. The spike records sold-payload attributes but is not explicitly told to record the *active-listing breadcrumbs* shape that drives tiering.

- **[LOW] Migration lacks a JSONB GIN/validation and `market_status` has no default; older rows read `null` status.** 03-02 adds `market_status text` with no default. Existing analyses (and new Phase-1 rows) will have `market_status = null`. 03-06's section state machine must treat `null` as "no market data yet → show fetch trigger," which it does implicitly, but the plan never states the null-status semantics the way BRF did (`brf_status null = upload affordance`). Minor, but a source of a "stuck/blank panel" bug if mishandled.

- **[LOW] The `prisPerKvm` divide-by-zero also exists on `areaAvg`.** If `comps` is non-empty but every comp's `prisPerKvm` normalizes to null (a real possibility per the null-tolerant schema), `areaAvg` is `0` or `NaN` and `deltaPct` is `Infinity`/`NaN`. The thin-data marker keys on `comps.length`, not on "comps with usable pris/kvm," so a set of attribute-only comps slips past the thin guard.

4. **Suggestions**

- (→ guest blind spot) In 03-05/03-06, explicitly state that Phase 3 panels are **owner-only** (inherit the persisted-row + RLS reality from Phase 1). Drop the `isGuest` read-only ambiguity in 03-06 Task 2, or define exactly what a guest sees (almost certainly: nothing, because there is no guest analysis row). Add a sentence to 03-CONTEXT/ROADMAP clarifying Success Criterion 3 is evaluated for an authenticated owner.
- (→ prisPerKvm=0) Add a RED test case in 03-01 Task 2: `listingPrisPerKvm <= 0` → comparison returns a "listing pris/kvm okänt" marker, NOT a -100% headline. In 03-05, guard: if `row.listing_data.prisPerKvm` is `0`/falsy, set `price = null` with an honest "pris/kvm saknas för objektet" reason rather than computing a bogus delta.
- (→ spike-failure honesty) Introduce a distinct `market_status: "source_unavailable"` (or a `priceData.reason: "no_source" | "thin"`) so 03-06 can show "Prisjämförelse är tillfälligt otillgänglig" (source down) vs "För få försäljningar" (genuinely thin). Add a phase-completion gate in 03-06 Task 3 (human-verify) that **fails the phase** if the price panel only ever renders against the mock fixture — i.e. require at least one real live sold comparison before marking PRICE-01 done.
- (→ deso.geojson) Make 03-03 Task 1 verify `ogr2ogr` availability FIRST and, if absent, commit to the explicit fallback (kommun-only baseline) as the *expected* path, not an afterthought. Quantify the committed GeoJSON size budget and consider simplifying polygons (tippecanoe/`mapshaper`) to keep the bundle small. State the AREA-01 degradation (kommun-only) honestly if DeSO conversion is not done in-session.
- (→ SoldComp ordering) Reorder 03-04: write sold-schema.ts (with `SoldComp`) in Task 1, compare.ts in Task 2 — or co-locate `SoldComp` in compare.ts. Fix the verify gate so a missing-module failure is not masked by `|| echo TYPES-OK`.
- (→ SCB DeSO availability) Add a live probe to 03-01 Task 1 (or a new 03-03 Task 1 sub-step): query the income + tenure tables at one DeSO code for the latest year and record actual availability. Set expectations in the plan accordingly (likely kommun-level for income/tenure).
- (→ rate limit / cost) In 03-05, bound the fetch walk itself: cap the tiered fallback at N total source calls per analysis (e.g. ≤3), short-circuit as soon as a tier yields enough comps, and add at minimum a coarse per-user/day enrich counter (even a simple column or count query) rather than deferring all rate-limiting. Persist the SCB results as the cache of record (use `area_data` or a dedicated cache table) so cold starts don't re-hit SCB.
- (→ areaAvg NaN) In 03-04, compute `areaAvg` only over comps with a finite `prisPerKvm`, and treat "0 usable comps" as thin/`Infinity`-safe (clamp/guard deltaPct). Add a RED test: comps present but all pris/kvm null → thin marker, no NaN.
- (→ market_status null) State the `market_status = null → fetch trigger` semantics explicitly in 03-02 and 03-06, mirroring how 002_brf documented `brf_status` states.

5. **Risk Assessment** — **MEDIUM**

The architecture is sound and the de-risking discipline is real, so this is not HIGH. But three HIGH concerns (guest blind spot, prisPerKvm=0 poisoning the headline, and dishonest spike-failure degradation that can pass all automated gates while shipping a broken/mislabeled panel) each independently threaten either Success Criterion 1 or the D-09 trust contract. None are architectural rewrites — all are fixable with targeted test cases, one new status value, an ordering fix, and explicit honesty about the kommun-only fallback. With those addressed, risk drops to LOW.
