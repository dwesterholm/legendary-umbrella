<objective>
Make the server-side discovery run finish inside the client poll window for a realistic multi-area query (300+ listings) by parallelizing area scrapes and bounding per-page render time — without weakening the shared cost/candidate caps. Covers DXUX-01 (backend throughput half).

Implements CONTEXT decisions D-01 (parallelize areas in runSlice), D-02 (scoped per-area-page waitSecs / retry override), and preserves the LOCKED D-03 cost-cap race-free invariant.

Purpose: today the areas in "Södermalm och Vasastan" scrape sequentially and a single timing-out page can block its area's batch ~180s, so a large query blows past the client's 5-min poll ceiling AND risks the 300s Vercel Server Action kill (RESEARCH Pitfalls 1–2). D-01 collapses sum(area times) → max(area time); D-02 caps what that max even is. Both must ship together (Pitfall 1).
Output: a parallelized `runSlice` area loop, an additive `opts` param on `runPlaywrightRender`, a scoped `AREA_PAGE_WAIT_SECS` override on `fetchAreaPage`, and the two Wave-0 tests (concurrency-timing proof + scoped-override assertion) from 13-VALIDATION.md.
</objective>
