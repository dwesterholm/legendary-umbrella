<objective>
Fix the discovery progress counter so it means "candidates analyzed / candidates found" (LOCKED operator decision 2026-07-22), and revert the 13-04 Task 2 regression that overwrote `processed_count` during the vision pass.

The 13-03 re-smoke surfaced a nonsensical counter: "350 av 25 annonser analyserade". Two root causes:
1. **Pre-existing:** `processed_count` accumulates TOTAL LISTINGS SCANNED during the area scrape (runSlice, reaching ~350), but discovery-progress.tsx displayed it as `${processedCount} av ${capCandidates}` (cap_candidates=25) → "350 av 25".
2. **Introduced by 13-04 Task 2:** the vision-phase `onProgress` callback overwrites `processed_count` with the enriched count (1..≤8), making the displayed counter jump BACKWARD (350 → 3).

Purpose: The counter must read the operator's locked semantics — numerator = candidates analyzed so far (monotonic, never exceeds the denominator), denominator = candidate_count (candidates actually found), reaching "N av N" at status `done`. `processed_count` keeps its scanned-listings scrape/cost meaning; it is never the counter's numerator and is never written by the vision pass.

Output: A corrected client counter (display-layer + minimal monotonic component-state — NO DB migration), plus a server-side revert removing the processed_count overwrite.
</objective>

<design_decision>
**No DB migration.** The operator guidance prefers a display-layer + minimal-state fix if a coherent "analyzed" numerator can be derived from existing row fields, and explicitly cautions against a new column (see project memory `supabase-migration-already-applied`).

A coherent numerator IS derivable from existing fields with zero schema change:
- Denominator = `candidate_count` (already selected + already in the row).
- Numerator ("analyzed") = a component-state value, monotonic non-decreasing, clamped to `candidate_count`. Its target is `candidate_count` when status is `done` (all found candidates have been scored — the important nuance: only `VISION_ENRICH_LIMIT`=8 candidates get deep detail-fetch/vision, but ALL `candidate_count` candidates are scored/shown, so "analyzed" at `done` = candidate_count), and 0 while the job is still `pending`/`processing`/`vision_processing` (nothing is committed as analyzed until the terminal write).

Consequence and tradeoff (documented on purpose): during the multi-minute `vision_processing` window the counter holds at "0 av N" — there is NO existing row field that can carry a persisted mid-run "analyzed" count without either corrupting `processed_count` (forbidden) or adding a new column (avoided). Run liveness during that window is still carried by the badge advancing to "Analyserar bilder" (the 13-04 decoupled read, preserved) and the soft-notice — so DXUX-01's "visibly progresses" signal is not lost, only the numeric counter is quiet until `done`, where it snaps to "N av N". A persisted mid-run numerator would require a new `analyzed_count` column (migration); it is explicitly out of scope for this fix per operator guidance and is not needed to satisfy any locked requirement.

D-03 (cost-cap invariant, LOCKED) is preserved by construction: this plan REMOVES a DB write (the vision-phase processed_count overwrite) and adds none. No cost/cap read-then-write (check-then-act) window is opened or reopened; the terminal `updateJob(status:"done", results)` write is unchanged.
</design_decision>
